# Threat Model

**Document:** `docs/10_Security/threat_model.md`  
**Version:** 1.0  
**Status:** Approved  
**Project:** Apartment Complaint Logging System (ACLS)

---

> This document identifies the primary threats to ACLS and the mitigations in place. It is structured using the STRIDE model (Spoofing, Tampering, Repudiation, Information Disclosure, Denial of Service, Elevation of Privilege). Any new feature or infrastructure change must be reviewed against this model before implementation.

---

## 1. System Trust Boundaries

```
[Client apps] ──HTTPS──► [ACLS.Api] ──SQL/TCP──► [Azure SQL]
                                    ──HTTPS──► [Azure Blob Storage]
                                    ──async──► [ACLS.Worker]
                                                    ──HTTPS──► [Email Provider]
                                                    ──HTTPS──► [SMS Provider]
```

Trust boundaries:
- **Internet → API:** All client traffic crosses this boundary. TLS required. JWT required.
- **API → Database:** Internal Azure VNet. Connection string from Key Vault.
- **API → Blob Storage:** Internal Azure. Connection string from Key Vault.
- **Worker → External Providers:** Outbound HTTPS. API keys from Key Vault.

## 2. STRIDE Threat Analysis

### 2.1 Spoofing (Identity)

| Threat | Component | Mitigation |
|---|---|---|
| Attacker impersonates a Resident by forging a JWT | ACLS.Api | HMAC-SHA256 signature validated on every request. Signing key in Key Vault — never in source code. |
| Attacker replays an expired JWT | ACLS.Api | `exp` claim validated. Token lifetime: 60 minutes. |
| Attacker registers with a forged invitation token | POST /auth/register | Token validated against DB: exists, not expired, not used, not revoked. |
| Attacker guesses another resident's password | POST /auth/login | BCrypt (work factor 12) makes brute force computationally impractical. No lockout in V1 — V2 concern. |

### 2.2 Tampering (Data Integrity)

| Threat | Component | Mitigation |
|---|---|---|
| Attacker modifies a complaint they did not submit | API + DB | `PropertyId` from JWT (not request). Resident can only access their own complaints. Repository filters by `PropertyId` and `ResidentId`. |
| Attacker modifies the `PropertyId` in a request to access another property | ACLS.Api | `PropertyId` is never accepted from request bodies. `TenancyMiddleware` reads it exclusively from the JWT claim. |
| Attacker tampers with media URLs to serve malicious content | Media table | URLs are generated by `IStorageService` after upload. Clients cannot set blob URLs directly. |
| Attacker injects SQL via complaint fields | ACLS.Persistence | EF Core parameterised LINQ queries. No raw SQL in V1. |

### 2.3 Repudiation (Audit)

| Threat | Component | Mitigation |
|---|---|---|
| User denies having submitted or modified a complaint | AuditLog | Immutable `AuditEntry` written for every critical action. Includes actor ID, role, IP address, timestamp, and before/after snapshot. |
| System-generated actions cannot be attributed | AuditLog | System events (Worker jobs) written with `ActorUserId = null` and `ActorRole = null` — clearly distinguishable. |

### 2.4 Information Disclosure

| Threat | Component | Mitigation |
|---|---|---|
| Resident A views Resident B's complaint | ACLS.Api | Repository filters by `PropertyId` + entity ownership. Cross-property requests return 404 (not 403) to prevent ID enumeration. |
| Manager at Property A views Property B's data | ACLS.Api | `TenancyMiddleware` + mandatory `PropertyId` filter on all queries. |
| Resident feedback visible to other residents | Complaints | `ResidentFeedbackComment` and `ResidentRating` never included in responses to Resident role (NFR-13). |
| Passwords leaked in logs | Logging | Password fields never logged. `POST /auth/login` request body never logged. |
| JWT tokens leaked in logs | Logging | JWT values never logged. Only `UserId` and `PropertyId` are logged from claims. |
| Internal error details leaked to clients | ExceptionHandlingMiddleware | 500 responses return generic `System.InternalError` code. Stack traces never serialised to HTTP responses. |

### 2.5 Denial of Service

| Threat | Component | Mitigation |
|---|---|---|
| Attacker floods API with requests | ACLS.Api | Azure App Service auto-scaling. WAF/Load Balancer in front of App Service (V1 basic, V2 enhanced). |
| Outage broadcast overwhelms notification provider | ACLS.Worker | Concurrent fan-out with `Task.WhenAll()`. Provider rate limits handled via retry logic in `NotificationService`. |
| Large file uploads exhaust storage | Media upload | 5MB per file, 3 files per complaint enforced before upload. |
| Database connection pool exhaustion | ACLS.Persistence | EF Core `DbContext` is `Scoped` (per-request). Connection pool managed by SQL Server driver. |

### 2.6 Elevation of Privilege

| Threat | Component | Mitigation |
|---|---|---|
| Resident calls Manager-only endpoint | ACLS.Api | `[Authorize(Roles = "Manager")]` on all manager endpoints. Role claim validated from JWT — not from request body. |
| Maintenance Staff assigns a complaint (Manager-only action) | ACLS.Api | `[Authorize(Roles = "Manager")]` on `/complaints/{id}/assign`. Staff role receives 403. |
| Staff sets their own availability to BUSY | POST /staff/{id}/availability | `BUSY` excluded from the allowed values in `UpdateAvailabilityRequest`. Returns `Staff.CannotSetBusyManually` (422) if attempted. |
| Manager accesses another property's data | ACLS.Api | `PropertyId` in JWT is set at login and cannot be changed by the client. Manager scoped to their own property. |

## 3. V2 Security Improvements (Out of Scope for V1)

- Account lockout after N failed login attempts
- Refresh token rotation
- Per-IP rate limiting
- Pre-signed blob URLs with short expiry (replace permanent URLs)
- Two-factor authentication for Manager accounts
- Penetration test findings documented in `docs/10_Security/penetration_test_findings/`
