SCOPE
-----
Phase 4a built the complete Next.js 14 Manager Dashboard frontend as an npm
workspace monorepo. Three shared packages were created alongside the web app:

  @acls/shared-types   — TypeScript enum mirror of C# backend enums
  @acls/api-contracts  — TypeScript interfaces for all request/response DTOs
  @acls/sdk            — Axios HTTP client with JWT Bearer interceptor
  apps/ResidentApp.Web — Next.js 14 App Router manager dashboard

No business logic lives on the client. All state flows from the ACLS.API backend
via the lib/api layer. Components are pure rendering and event delegation.

KEY DECISIONS
-------------

1. next.config.mjs (not .ts)
   Next.js 14 does not support next.config.ts (that was introduced in v15).
   The file was named next.config.ts during initial generation; it was renamed
   to .bak and replaced with next.config.mjs before the build.

2. OutageType enum casing
   v1.yaml defines OutageType values as PascalCase (Electricity, Water, Gas, …).
   The TypeScript enum uses those exact string values — not SCREAMING_SNAKE_CASE —
   so the frontend payload matches what the API sends and accepts.

3. transpilePackages
   @acls/* packages ship TypeScript source, not compiled JS. next.config.mjs sets
   transpilePackages so Next.js compiles them inline — no separate tsc build step
   needed for workspace packages.

4. design_system.md compliance (retroactive)
   docs/08_UX/design_system.md was added after Phase 4a was originally completed.
   A compliance pass was applied on 2026-03-20 to align colours, icons, layout,
   and component behaviour with the design system. See COMPLIANCE PASS section.

5. Auth strategy
   JWT stored in localStorage under key "acls_access_token". The SDK interceptor
   attaches it as Bearer on every outgoing request. On 401, token is cleared and
   browser is redirected to /login. No server-side cookie / middleware auth was
   added (all pages use client-side isAuthenticated() check via useEffect).

6. No business logic on client
   Per ai_collaboration_rules.md Rule 1 (thin frontends): components contain only
   rendering and user-event delegation. All data transformation and validation is
   handled by the backend. The frontend trusts API responses directly.

SECURITY NOTE
-------------
next@14.2.35 still carries high-severity CVEs affecting all versions ≤15.5.13:
  - DoS via Image Optimizer remotePatterns (GHSA-9g9p-9gw9-jx7f)
  - HTTP request deserialization DoS with React Server Components (GHSA-h25m-26qc-wcjf)
  - HTTP request smuggling in rewrites (GHSA-ggv3-7p47-pfv8)
  - Unbounded next/image disk cache growth (GHSA-3x4c-7xq6-9pq8)

npm audit fix --force would upgrade to next@16.x (breaking changes). Upgrading
to Next.js 15 or 16 is recommended before production deployment.
