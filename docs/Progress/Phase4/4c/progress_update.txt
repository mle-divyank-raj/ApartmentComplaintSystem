PHASE 4c PROGRESS UPDATE
========================
Date: Phase 4c completion
Focus: iOS Native Applications — ResidentApp.iOS and StaffApp.iOS

OVERVIEW
--------
Phase 4c delivered complete iOS native implementations for both the resident-facing and
staff-facing apps, following the same architecture patterns established in Phase 4a (Android
ResidentApp) and Phase 4b (Android StaffApp).

Both apps use:
  - Swift 5.0 + SwiftUI, iOS 16.0+ deployment target
  - async/await exclusively (no Combine, no completion handlers)
  - @MainActor final class ViewModels with @Published private(set) state
  - JSONDecoder with .convertFromSnakeCase + .iso8601 date strategy
  - JWT stored in iOS Keychain via TokenStore
  - Base URL read from Info.plist key API_BASE_URL (never hardcoded)
  - NavigationPath-based routing via NavigationRouter + AppRoute/StaffAppRoute enums
  - All error codes mapped to NSLocalizedString keys via ErrorCodeMapper

TECHNICAL NOTES
================================================================================

1. NO SHARED MODULE: Both apps contain their own copy of APIClient, Enums, TokenStore, 
   ErrorCodeMapper. This matches the Android pattern and avoids a cross-platform dependency.

2. MULTIPART UPLOAD: Built manually with boundary string in APIClient. No third-party 
   dependencies.

3. PHOTO PICKER: Uses PhotosUI.PhotosPicker (iOS 16+). Data loaded via 
   loadTransferable(type: Data.self) and passed as MultipartFile array.

4. PARAMETER INJECTION: TokenStore is passed as a parameter to ViewModel methods rather 
   than injected at init, to avoid @StateObject / @EnvironmentObject init-time conflicts.

5. BUSY STATE: Staff cannot manually set BUSY — only AVAILABLE, ON_BREAK, OFF_DUTY are 
   offered in UpdateAvailabilityView. Footer explains BUSY is set automatically.

6. SOS FLOW: Three-step: warning screen → native .confirmationDialog → form (title 
   pre-filled "Emergency") → confirmed screen showing ASSIGNED status.
